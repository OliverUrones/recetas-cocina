Proyecto en Grupo
Descripción del Trabajo a Realizar / Entregar.

   El trabajo a realizar va a considerarse un trabajo en grupo, donde todos y cada uno de vosotros formáis parte de un equipo de trabajo que debe desarrollar un proyecto completo.
   Dentro del equipo habrá subgrupos a los que se les asignarán tareas de desarrollo diferentes, pero todas ellas relacionadas con el mismo proyecto final.

   El objetivo será...
   Desarrollar una Aplicación Web completa con el framework "CakePHP".
   Suponed que nos han encargado un trabajo de desarrollo en el que debemos realizar una aplicación web que sirva para realizar una determinada gestión.

   La especificación básica de la gestión a realizar y otros archivos necesarios, están adjuntos a esta descripción, y son los siguientes:
  1. Requisitos Iniciales (archivo ".html")
  2. Análisis Inicial de Datos y Tareas (archivo ".html")
  3. Diseño Inicial de la Base Datos (archivo ".mwb")
  4. Esquema Inicial de la Base Datos (archivo ".sql")
  5. Grupos de Trabajo y Tareas asignadas (archivo ".html")
   Con esto, ya se puede empezar a trabajar en la aplicación, aunque primeramente deberéis repartiros el trabajo entre los grupos que hayáis formado.

   Como todos estaréis colaborando y programando a la vez en el mismo proyecto, estaréis de acuerdo en que se deben compartir los archivos fuente de alguna forma (Control de Versiones tipo SVN, GIT, ...) para que todos tengáis acceso al código fuente del proyecto para modificarlo y hacer las pruebas correspondientes.

   Además, como la aplicación utiliza una Base de Datos para almacenar la información de la misma, el diseño inicial se os proporciona en un archivo desarrollado con "MySQL Workbench 6.3" el cual podéis analizar para entender la estructura de los datos y el diseño propuesto, y aunque también se incluye el archivo ".sql" con las órdenes para la base de datos, podéis usar el diseño inicial para crearlo o sincronizarlo con vuestro servidor de Base de Datos.

Otras Notas:
  • Este trabajo se realizará en grupos de 2 personas. Si sóis impares, se permitirá que uno sea individual, o quizá un grupo de 3 personas.
  • El proyecto deberá estar terminado y funcionando en la fecha indicada (mediados de Enero aproximadamente), y se utilizarán los registros del Sistema de Control de Versiones utilizado para analizar el trabajo realizado por cada grupo de trabajo, y en concreto por cada uno de vosotros.
  • Independientemente del Sistema de Control de Versiones utilizado, se habilitará una tarea para que cada uno de los grupos suba su parte de trabajo realizado (fuentes desarrollados), que aunque es difícil diferenciarlo cuando se trabaja en equipo, se ha intentado que los bloques de trabajo diferencien perfectamente los archivos desarrollados y/o carpetas de la aplicación.
  • Importante: La gestión web que se realizará va a tener una parte pública y una privada, y ambas utilizarán el framework para proporcionar su funcionalidad. No se admitirán partes en el desarrollo hechas sin usar el propio framework o alguna de sus funcionalidades / utilidades, ni realizadas de forma que no se adapten al uso normal del mismo, incluyendo aquí el que las partes programadas se puedan conseguir hacer de manera equivalente aprovechando lo que ya está implementado en el framework.
  • Obligatorio: Al menos deberá tener 5 roles para los permisos de acceso (3 propios de la aplicación desarrollada, 1 como Usuario Invitado y 1 como Usuario Administrador de Sistema), aunque pueden surgir más si se consideran necesarios, como por ejemplo, que se separen algunas de las funciones de un tipo de usuario en varias tareas dentro de la aplicación, o algo similar, con lo que se debe dejar preparada la aplicación para que se puedan agregar más funciones y opciones de acceso a los nuevos perfiles. En la gestión de asignación de roles a usuarios de la aplicación es necesario que todos tengan al menos un rol, a excepción del Usuario Invitado que no es un usuario autenticado aún en el sistema. Y no es necesario que un usuario pueda tener más de un rol asignado, a excepción del Administrador del Sistema que podrá hacer cualquier cosa en la aplicación, e incluso actuar como un usuario que tenga un rol determinado.
